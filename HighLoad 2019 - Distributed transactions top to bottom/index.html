WHY?

  • MongoDB marketing: 80% - 90% of apps don't need transactions.
  • Henrik: Well, maybe 50%...
  • Werner Vogels, on Amazon.com:
    • About 70% of operations were of the key-value kind, where only a primary key was used and a single row would be returned. About 20% would ... still operate on only a single table.

Multi Version Concurrency Control

  • Bernstein & Goodman, 1981. Reed, 1979. InterBase, 1984.
  • Updates create new versions of records
  • All reads from same version
  • Updates to same record cause conflict

Write Conflict

> session1 = db.getMongo().startSession()
> accounts = session1.getDatabase("examples").accounts
> session1.startTransaction()
> accounts.find()
{ "_id" : 1, "balance" : 100 }
{ "_id" : 2, "balance" : 50 }
> accounts.update({ _id:1}, {$set: {balance:100-10} })
WriteResult({ "nMatched" : 1, "nUpserted" : 0, "nModified" : 1 })

                
> session2 = db.getMongo().startSession()
> accounts = session2.getDatabase("examples").accounts
> session2.startTransaction()
> accounts.find()
{ "_id" : 1, "balance" : 100 }
{ "_id" : 2, "balance" : 50 }
> accounts.update({ _id:1}, {$inc: {balance:100-11} })
WriteCommandError({
        "errorLabels" : ["TransientTransactionError"],
        "operationTime" : Timestamp(1572204799, 5),
        "errmsg" : "WriteConflict",
        "code" : 112,   ...
})
> session2.abortTransaction()
 
> accounts.find()
{ "_id" : 1, "balance" : 90 }
{ "_id" : 2, "balance" : 50 }
> session1.commitTransaction()
> db.accounts.find()
{ "_id" : 1, "balance" : 90 }
{ "_id" : 2, "balance" : 50 }
  • Consistency is preserved!
